# üö® MASTER INDEX ADDENDUM ‚Äî PERSONAL INJURY (DISTRIBUTION VERTICAL & PHASE GOVERNANCE)

**Status:** LOCKED (Append-Only)

## I. STRATEGIC CLASSIFICATION (AUTHORITATIVE)

Personal Injury (PI) is a **Distribution Vertical**.

PI is explicitly excluded from Phase-1 sponsorship monetization and is not evaluated under the same criteria as TRT, Dentistry, USCIS, or other authority-based verticals.

PI advertisers purchase **control over intake and case flow**, not ‚Äúauthority sponsorship.‚Äù Therefore, PI ‚Äúbuyouts‚Äù are classified as **distribution contracts**, not sponsorships.

## II. MONETIZATION MODE ENUM (CANONICAL LAW)

The PI vertical MUST support a governance-level monetization mode enum:

* **EDUCATION_ONLY**
* **DISTRIBUTION_SPONSORED_DIRECTORY**
* **FULL_SITE_SPONSORSHIP**

A city‚Äôs monetization mode MUST be explicitly declared. Absence of a monetization mode is a validation failure.

## III. PHASE GOVERNANCE (LOCKED)

### Phase 1 ‚Äî Foundation (Non-Monetized)

* Build and validate distribution scaffolding
* All cities remain EDUCATION_ONLY
* No sponsor-gated mechanisms permitted

### Phase 2 ‚Äî Distribution (Primary PI Monetization Phase)

* Allowed monetization mode: DISTRIBUTION_SPONSORED_DIRECTORY
* Sponsored directory behavior is permitted only in Phase 2
* Sponsor-gated Next-Steps mechanisms are permitted only in Phase 2

**Phase-2 Intake Cities (Authorized Set):**

* Houston
* Chicago
* Atlanta
* Orlando
* Dallas

### Phase 3 ‚Äî Full-Site Sponsorship (Optional / Non-Primary)

* FULL_SITE_SPONSORSHIP may be offered as an optional product
* Full-site sponsorship pricing may remain in the system (e.g., $75K), regardless of likelihood
* Phase 3 does not override Phase-2 distribution constraints unless explicitly amended

## IV. SPONSORED DIRECTORY BEHAVIOR (PHASE-2 LOCKED)

### Sponsored Placement Rules

When a city is in DISTRIBUTION_SPONSORED_DIRECTORY and sponsor state is live:

* The city directory remains present
* A single sponsored firm block renders at the top of the directory
* A CTA may appear only inside the sponsored firm block

### Directory Integrity Rules

* The directory MUST NOT be removed, hidden, or replaced
* Neutral directory entries remain present below the sponsored block
* No ranking, rating, or comparative ordering is permitted

## V. DISCLOSURE REQUIREMENTS (NON-NEGOTIABLE)

When a sponsored directory block is present, disclosure MUST appear immediately above it.

Disclosure MUST be concise and canonical (see Appendix H).

## VI. VALIDATION RULES (POLICY-LEVEL)

Validation MUST fail if:

* A city is missing an explicit monetization mode
* A sponsored block appears in EDUCATION_ONLY
* More than one sponsor appears live in a city
* A CTA appears outside the sponsored block
* Sponsor-gated Next-Steps exists when sponsor is not live
* Sponsor is live in Phase 2 but required sponsor surfaces are missing

## VII. GOVERNANCE GUARANTEE

PI distribution behavior MUST NOT be reused across verticals without an explicit addendum update.

# SECTION X ‚Äî SCALABLE SPONSOR + NEXT-STEPS ARCHITECTURE (ALL VERTICALS)

## Core Principle (Do Not Fork)

Do NOT copy/paste PI logic into other verticals.
Generalize the mechanism once in the base generator, then enable/shape it per vertical via pack config + per-city data.

## 1) Make Sponsor/Next-Steps Vertical-Agnostic (One-Time Engine Work)

**Goal:** Same engine; different copy/placement rules per vertical.

### Standard Sponsor Object Schema (Canonical Input Contract)

Use one sponsor object shape across all verticals (per-city truth lives in the city listing JSON):

Sponsor object:

* status: "live"
* firm_name: "..."
* official_site_url: "https://..."
* intake_url: "https://..."

### Generic Next-Steps Route Generator

* Create //next-steps/ ONLY when sponsor.status = live AND the vertical‚Äôs pack allows next-steps.
* Validation must FAIL if:

  * next-steps exists but sponsor is not live
  * sponsor is live (and next-steps is enabled) but next-steps is missing

### Generic CTA Injection Hook

* CTA rendering must be configurable per vertical (NOT hard-coded to PI).
* CTA placement options must be pack-driven (see ¬ß2).

## 2) Put Vertical-Specific Behavior in the Page-Set Pack (No Build Forks)

Each vertical pack (e.g., dentistry_v1.json, trt_v1.json, etc.) must own a sponsorship config block that controls:

* whether next-steps is enabled for this vertical
* where CTA appears (city hub only, directory module only, both)
* button copy (registry-backed)
* disclosure copy (registry-backed)
* directory behavior (embedded vs separate page, required vs optional)
* styling variant (table vs cards)

Result: Dentistry can use a different UX (e.g., ‚ÄúRequest an appointment‚Äù) without inheriting PI intake language.

## 3) Keep Per-City Sponsor Truth in data/listings/.json

Operational scaling rule:

* Sales ops flips sponsor live status by editing ONE city file.
* Build output reflects it automatically.
* Validator prevents accidental leakage.

Dentistry/TRT/etc. can use the same sponsor object when a sponsor is sold‚Äîno new mechanism code.

## 4) Use data/sponsors/programs.json as the Product Catalog (Not Live Status)

Treat programs.json as a catalog of what can be sold and how it displays:

* what sponsorship products exist (directory sponsor, city sponsor, full-site buyout)
* what pages/slots each product affects
* what copy appears on /for-providers/
* pricing tables + inclusions

Critical separation:

* programs.json defines products + display.
* city listing JSON controls who is live.

## 5) Validation Strategy That Stays Maintainable

Validator design:

* Base validator checks universal invariants:

  * next-steps exists ‚áÑ sponsor live (when enabled)
  * disclosure proximity
  * no multiple sponsors per city
* Per-vertical rule set loads from pack config:

  * is next-steps enabled?
  * is CTA allowed on city hub? directory? both?
  * is directory embedded required/optional?

This preserves PI strictness without forcing other verticals into PI UX.

## Fastest Path Later (Lowest Work Sequence)

1. Extract PI sponsor/next-steps logic into a reusable module in build_city_sites.js (segment, don‚Äôt fork).
2. Add sponsorship config block to each pack JSON.
3. Update validator to read pack config and apply correct rule set.
4. Dentistry rollout becomes data-only: add sponsor stubs to dentistry cities + set ONE city sponsor live to test.
